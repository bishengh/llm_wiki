## 背景

当前的LLM文档交互系统依赖于RAG技术，每次查询都会从头重新发现知识，而不会积累理解。这限制了跨多个文档合成信息和构建连贯知识库的能力。提议的解决方案是实现一个持久化的wiki系统，逐步构建和维护结构化、相互链接的markdown文件集合。

## 目标/非目标

**目标：**
- 创建一个持久化的wiki系统，随时间构建和维护知识
- 支持将新源增量集成到现有wiki中
- 开发更新实体页面和主题摘要的机制
- 建立用户与wiki交互的工作流程
- 确保wiki保持一致和更新
- 实现系统维护功能，包括内容验证、性能优化和备份恢复

**非目标：**
- 完全替换现有的RAG系统
- 处理wiki内容的实时协作
- 支持超越基本历史记录的复杂版本控制
- 实现高级访问控制系统

### 2. 核心模块

- **Wiki创建模块**：负责wiki初始化、目录结构创建、默认页面生成和配置管理

- **知识集成模块**：管理信息集成、实体和主题管理、链接管理和冲突解决
- **Wiki交互模块**：处理自然语言查询、页面管理和用户反馈收集
- **Wiki维护模块**：执行内容验证、性能优化和备份恢复

## 决策

### 1. Wiki存储结构
- **决策**：使用分层文件系统和markdown文件存储wiki内容
- **理由**：Markdown是人类可读的、可版本控制的，并且得到广泛支持。文件系统结构允许轻松组织和导航。
- **考虑的替代方案**：数据库存储（更复杂，可移植性较差）和专用wiki格式（标准度较低）

### 2. 知识提取和集成
- **决策**：使用LLM执行提取和集成任务
- **理由**：LLM非常适合理解和合成来自各种来源的信息。使用同一个LLM执行这两个任务确保处理的一致性。
- **考虑的替代方案**：用于提取的单独NLP工具（灵活性较低）和人工管理（可扩展性问题）

### 3. 增量更新策略
- **决策**：实现基于差异的wiki页面更新方法
- **理由**：这允许高效更新并保持清晰的变更历史。它还有助于识别冲突并需要最小的重新处理。
- **考虑的替代方案**：完整页面再生（效率低下）和手动更新（可扩展性问题）

### 4. 用户交互模型
- **决策**：创建基于聊天的界面用于wiki交互和更新
- **理由**：聊天界面对用户来说很熟悉，允许自然语言请求。它们还为LLM理解用户意图提供上下文。
- **考虑的替代方案**：传统wiki编辑（用户友好性较低）和命令行界面（可访问性有限）

### 5. 存储和性能考虑
- **决策**：使用本地文件系统存储，可选云同步
- **理由**：本地存储确保离线访问和简单性，而云同步启用备份和多设备访问。
- **考虑的替代方案**：完全云存储（依赖互联网连接）和数据库存储（额外复杂性）

## 实现细节

### 1. 目录结构

```
src/
├── config/          # 配置文件
├── interaction/     # 交互模块
├── knowledge/       # 知识集成模块
├── maintenance/     # 维护模块
├── source/          # 源集成模块
├── wiki/            # wiki存储
│   ├── entities/    # 实体页面
│   ├── topics/      # 主题页面
│   ├── attachments/ # 附件
│   ├── history/     # 历史记录
│   └── templates/   # 模板
├── public/          # 前端文件
├── main.js          # 主服务器文件
├── package.json     # 项目配置
└── uploads/         # 上传文件临时存储
```

### 2. API设计

- **POST /api/wiki/init** - Wiki初始化
- **POST /api/files/upload** - 文件上传
- **POST /api/files/upload-batch** - 批量文件上传
- **POST /api/knowledge/integrate** - 信息集成
- **POST /api/interaction/query** - 自然语言查询
- **GET /api/pages** - 列出页面
- **GET /api/pages/:path** - 获取页面内容
- **PUT /api/pages/:path** - 更新页面内容
- **GET /api/search** - 搜索功能
- **POST /api/feedback** - 反馈收集
- **POST /api/maintenance/run** - 运行维护
- **POST /api/maintenance/backup** - 创建备份
- **GET /api/maintenance/backups** - 列出备份
- **POST /api/maintenance/restore** - 从备份恢复

## 风险/权衡

### 1. 知识一致性风险
- **风险**：LLM在集成新信息时可能引入不一致或错误
- **缓解措施**：实施验证检查，维护变更历史，并允许用户更正

### 2. 处理开销风险
- **风险**：处理大型文档或频繁更新可能在计算上昂贵
- **缓解措施**：实施批处理、节流和可选的后台处理

### 3. 存储增长风险
- **风险**：wiki可能无限增长，导致存储和性能问题
- **缓解措施**：实施修剪策略、压缩和选择性归档

### 4. 集成复杂性风险
- **风险**：集成不同的文档格式和来源可能很复杂
- **缓解措施**：从支持的格式开始，逐步扩展，为不同的源类型使用适配器

### 5. 用户采用风险
- **风险**：用户可能不愿从熟悉的基于RAG的系统切换
- **缓解措施**：提供渐进的过渡路径，展示明确的好处，并保持向后兼容性
